### Задание 1. СУБД

### Кейс
Крупная строительная компания, которая также занимается проектированием и девелопментом, решила создать 
правильную архитектуру для работы с данными. Ниже представлены задачи, которые необходимо решить для
каждой предметной области. 

Какие типы СУБД, на ваш взгляд, лучше всего подойдут для решения этих задач и почему? 
 
1.1. Бюджетирование проектов с дальнейшим формированием финансовых аналитических отчётов и прогнозирования рисков.
СУБД должна гарантировать целостность и чёткую структуру данных.

1.1.* Хеширование стало занимать длительно время, какое API можно использовать для ускорения работы? 

1.2. Под каждый девелоперский проект создаётся отдельный лендинг, и все данные по лидам стекаются в CRM к 
маркетологам и менеджерам по продажам. Какой тип СУБД лучше использовать для лендингов и для CRM? 
СУБД должны быть гибкими и быстрыми.

1.2.* Можно ли эту задачу закрыть одной СУБД? И если да, то какой именно СУБД и какой реализацией?

1.3. Отдел контроля качества решил создать базу по корпоративным нормам и правилам, обучающему материалу 
и так далее, сформированную согласно структуре компании. СУБД должна иметь простую и понятную структуру.

1.3.* Можно ли под эту задачу использовать уже существующую СУБД из задач выше и если да, то как лучше это 
реализовать?

1.4. Департамент логистики нуждается в решении задач по быстрому формированию маршрутов доставки материалов 
по объектам и распределению курьеров по маршрутам с доставкой документов. СУБД должна уметь быстро работать
со связями.

1.4.* Можно ли к этой СУБД подключить отдел закупок или для них лучше сформировать свою СУБД в связке с СУБД 
логистов?

1.5.* Можно ли все перечисленные выше задачи решить, используя одну СУБД? Если да, то какую именно?

*Приведите ответ в свободной форме.*

---

1.1. Для бюджетирования проектов и формирования финансовых аналитических отчетов идеально подойдет реляционная СУБД, такая как PostgreSQL или MySQL. Они обеспечивают целостность и четкую структуру данных, поддерживают транзакции и обладают мощными средствами для аналитических запросов.

1.1.* Для ускорения хеширования можно использовать API, такие как Google Guava или Apache Commons Codec. Они предлагают более эффективные алгоритмы хеширования, которые могут значительно ускорить процесс.

1.2. Для лендингов и CRM подойдет NoSQL СУБД, такая как MongoDB. Она обеспечивает гибкость структуры данных и быстрое чтение/запись, что идеально подходит для динамичных веб-страниц и работы с большим объемом данных.

1.2.* Да, задачу можно закрыть одной СУБД. MongoDB может быть использована как для лендингов, так и для CRM. Она обеспечивает гибкость и быстроту, что идеально подходит для этих задач.

1.3. Для создания базы данных по корпоративным нормам и правилам подойдет реляционная СУБД, такая как MySQL или PostgreSQL. Они обеспечивают четкую и понятную структуру данных.

1.3.* Да, можно использовать уже существующую СУБД из задач выше. PostgreSQL или MySQL могут быть использованы для этой задачи. Для реализации этого можно создать отдельную таблицу или базу данных в рамках существующей СУБД.

1.4. Для логистических задач подойдет графовая СУБД, такая как Neo4j. Она идеально подходит для работы со связями и позволяет быстро формировать маршруты.

1.4.* Да, к этой СУБД можно подключить отдел закупок. Neo4j позволяет быстро и эффективно работать со связями, что может быть полезно для отдела закупок.

1.5.* В теории, можно использовать только одну СУБД, но на практике это может быть неудобно и неэффективно. Например, PostgreSQL может быть использован для решения всех вышеупомянутых задач, но для некоторых из них (например, для логистических задач) более подходящими будут другие типы СУБД.

### Задание 2. Транзакции

2.1. Пользователь пополняет баланс счёта телефона, распишите пошагово, какие действия должны произойти для того, чтобы 
транзакция завершилась успешно. Ориентируйтесь на шесть действий.

2.1.* Какие действия должны произойти, если пополнение счёта телефона происходило бы через автоплатёж?

*Приведите ответ в свободной форме.*

---

Для успешного завершения транзакции по пополнению баланса счета телефона, следующие действия должны произойти:

1. Выбор способа пополнения: Пользователь должен выбрать удобный для себя способ пополнения баланса телефона, например, через банковскую карту, мобильное приложение или платежную систему.

2. Ввод суммы пополнения: Пользователь должен указать желаемую сумму пополнения баланса телефона.

3. Проверка данных: Система должна проверить введенные пользователем данные, такие как номер телефона и сумму пополнения, на корректность и соответствие формату.

4. Аутентификация пользователя: При необходимости система может потребовать аутентификации пользователя, например, ввод пароля или использование биометрических данных.

5. Подтверждение платежа: Пользователь должен подтвердить свое намерение пополнить баланс счета телефона, например, нажатием кнопки "Подтвердить" или вводом специального кода.

6. Обновление баланса: После успешного подтверждения платежа система должна обновить баланс счета телефона, увеличив его на указанную сумму пополнения.

Все эти шаги в совокупности обеспечивают успешное завершение транзакции по пополнению баланса счета телефона.

### Задание 3. SQL vs NoSQL

3.1. Напишите пять преимуществ SQL-систем по отношению к NoSQL. 

3.1.* Какие, на ваш взгляд, преимущества у NewSQL систем перед SQL и NoSQL.

*Приведите ответ в свободной форме.*

---

1. Структурированные данные: SQL базы данных предназначены для хранения структурированных данных, что означает, что данные организованы в таблицы, которые содержат строки и столбцы с определенными типами данных. Это позволяет легко определить и контролировать структуру данных, что упрощает запросы и анализ данных.

2. Гибкий язык запросов: SQL предоставляет мощный и гибкий язык запросов, который позволяет выполнять сложные операции, такие как объединение таблиц, агрегирование данных, сортировку и фильтрацию. Это делает SQL идеальным выбором для сложных аналитических запросов и отчетов.

3. ACID-транзакции: SQL базы данных поддерживают ACID-транзакции (Atomicity, Consistency, Isolation, Durability), что обеспечивает надежность и целостность данных. ACID-транзакции гарантируют, что либо все операции в транзакции будут выполнены успешно, либо ни одна из них не будет выполнена, что является важным для приложений, где целостность данных критически важна.

4. Поддержка сложных отношений: SQL базы данных легко поддерживают сложные отношения между данными, такие как связи "один-ко-многим" и "многие-ко-многим". Это позволяет эффективно моделировать и хранить данные, которые имеют сложные взаимосвязи, такие как связи между пользователями и их заказами или связи между студентами и курсами.

5. Масштабируемость: SQL базы данных обычно обладают хорошей масштабируемостью, особенно при использовании реляционных баз данных с горизонтальным масштабированием. Это позволяет обрабатывать большие объемы данных и поддерживать высокую производительность даже при увеличении нагрузки.

В целом, SQL базы данных предоставляют надежность, гибкость и эффективность для хранения и обработки структурированных данных, что делает их превосходным выбором для множества приложений.

### Задание 4. Кластеры

Необходимо производить большое количество вычислений при работе с огромным количеством данных, под эту задачу 
выделено 1000 машин. 

На основе какого критерия будете выбирать тип СУБД и какая модель *распределённых вычислений* 
здесь справится лучше всего и почему?

*Приведите ответ в свободной форме.*

---
При выборе типа СУБД и модели распределенных вычислений для работы с огромным количеством данных на 1000 машинах, следует учитывать несколько критериев:

1. Масштабируемость: Важно, чтобы выбранная СУБД и модель распределенных вычислений могли эффективно масштабироваться на такое большое количество машин. Это позволит обеспечить высокую производительность и обработку больших объемов данных.

2. Распределение данных: Учитывая, что данные будут храниться и обрабатываться на 1000 машинах, необходимо выбрать СУБД и модель распределенных вычислений, которые позволяют эффективно распределить данные между узлами сети. Это поможет достичь баланса нагрузки и увеличить скорость обработки данных.

3. Отказоустойчивость: При таком большом количестве машин неизбежными являются отказы узлов. Поэтому выбранная СУБД и модель распределенных вычислений должны обладать механизмами автоматического восстановления и репликации данных, чтобы минимизировать потерю информации и обеспечить непрерывную работу системы.

4. Соответствие требованиям задачи: Необходимо учитывать специфику задачи, с которой будет работать система. В зависимости от типа данных и операций, которые необходимо выполнять, могут быть предпочтительны различные типы СУБД и моделей распределенных вычислений.

На основе этих критериев можно рассмотреть несколько вариантов, которые могут быть подходящими:

1. Apache Hadoop и HDFS: Hadoop предоставляет распределенную обработку данных и хранение на основе HDFS (Hadoop Distributed File System). Он обладает высокой масштабируемостью, позволяет распределить данные между машинами и имеет механизмы автоматического восстановления. Hadoop также поддерживает различные модели обработки данных, такие как MapReduce, Spark, Hive и другие.

2. Apache Cassandra: Cassandra является распределенной СУБД, которая обеспечивает высокую производительность при работе с большими объемами данных. Она хорошо масштабируется и обладает отказоустойчивостью. Cassandra также поддерживает горизонтальное масштабирование и репликацию данных.

3. Apache Spark: Spark предоставляет распределенную обработку данных в памяти и поддерживает различные операции и модели обработки данных. Он обладает высокой производительностью и масштабируемостью, а также имеет механизмы восстановления данных и отказоустойчивости.

Выбор конкретного типа СУБД и модели распределенных вычислений будет зависеть от требований задачи, доступных ресурсов и экспертизы команды разработчиков.
